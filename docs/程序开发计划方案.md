# 《失落的女神》程序开发计划方案（双人并行）

> 一个老人追寻失落女神的旅程，最终发现自己寻找的不是神明，而是被遗忘的勇气。

- **游戏类型**：类绣湖（Rusty Lake / Cube Escape）点击式 2D 叙事解谜 —— **但主角老人在场景内可行走**（点击寻路，见下"主角操控"）
- **引擎**：Unity 2022.3 LTS（本机已装 `2022.3.62f3`，位于 `D:\unity\2022.3.62f3\Editor\Unity.exe`）
- **版本控制**：Git + Gitee/GitHub（含 Git LFS）
- **团队**：5 人 —— 程序 ×2（本方案对象）、策划、美术、文案
- **节奏**：每章 10 天一个周期，先做**序章**
- **章节**：序章 → 老年章 → 中年章 → 青年章

---

## 一、核心理念：为什么绣湖类适合双人并行

绣湖类游戏的本质是 **"一堆相对独立的房间/场景 + 若干可交互物件 + 谜题状态机 + 全局存档/循环"**。
这种结构天然适合把代码拆成两层：

- **系统层**（横向、全局、一次做好长期复用）
- **内容层**（纵向、每个房间/谜题一份、可无限并行）

只要**系统层的接口先约定好**，两位程序员就能各做各的房间/谜题，几乎不踩对方代码。这就是本方案分工的核心逻辑。

---

## 二、分工模式（推荐：系统层 A + 内容层 B）

| 角色 | 代号 | 负责 | 一句话职责 |
|------|------|------|-----------|
| 程序员 A | **系统程序** | 底层框架、存档、场景切换、时间循环、交互基类、音频/UI 管理器 | "搭舞台、定规则" |
| 程序员 B | **内容程序** | 具体房间、可交互物件、谜题逻辑、剧情触发、过场 | "在舞台上演戏" |

### 为什么这样分而不是"一人一半房间"
- 序章阶段系统层还没成型，若两人都做房间，会同时抢着改 GameManager、存档等公共文件 → **天天冲突**。
- 先由 A 把地基和"积木接口"打好，B 立刻能用这些积木拼房间，互不干扰。
- **中后期自然过渡到并行**：系统层稳定后，A 也可以下场做房间，两人各包不同房间，纯内容并行，零冲突。

### 阶段性并行度
```
序章前 3 天：  A 打地基       B 做美术占位/工具/第一个交互原型（弱依赖）
序章中 4 天：  A 补系统       B 大量产出房间与谜题（强并行）
序章后 3 天：  A B 各包剩余房间 + 联调打磨（完全并行）
```

## 三、代码架构总览

采用「**管理器（Manager）+ 场景（Scene）+ 可交互物件（Interactable）+ 谜题（Puzzle）+ 全局状态（GameState）**」五件套。

```
┌─────────────────────────────────────────────┐
│                  系统层 (A)                    │
│  GameManager   ── 全局入口、章节/循环调度         │
│  SceneLoader   ── 场景/房间加载与切换（含转场）    │
│  SaveSystem    ── 存档读写（JSON）               │
│  GameState     ── 全局标志位/物品/进度（可存档）    │
│  InventorySystem ─ 背包与物品使用（选中→点目标）   │
│  PlayerController ─ 主角老人行走（点击寻路/单屏可走带）│
│  CloseupView   ── 特写视图（抽屉/信件细看，不换场景）│
│  AudioManager / UIManager / InputManager        │
│  ── 提供给 B 的基类 ──                           │
│  InteractableBase   （可点击物件基类，点击→走到交互点→触发）│
│  PuzzleBase         （谜题基类，含解开事件）        │
│  WalkableArea       （房间可走区，B 每房摆一个）    │
│  DialogueSystem     （文案/对白播放）             │
└─────────────────────────────────────────────┘
                     ↑ 只依赖接口
┌─────────────────────────────────────────────┐
│                  内容层 (B)                    │
│  Room_00_Entrance / Room_01_xxx ...            │
│  各种 Interactable 派生（门、抽屉、符号、道具）     │
│  各章谜题 Puzzle 派生（拼图、序列、密码…）          │
│  剧情触发器、过场 Timeline                        │
└─────────────────────────────────────────────┘
```

### 关键接口约定（A 先交付，B 依赖）——这是双人不冲突的命脉
```csharp
// 1. 全局状态：一切进度都存这里，B 不要自己造存档
public static class GameState {
    public static bool GetFlag(string key);
    public static void SetFlag(string key, bool value);
    public static void AddItem(string itemId);
    public static bool HasItem(string itemId);
    public static void RemoveItem(string itemId);
}

// 2. 可交互物件基类：B 的所有可点击物都继承它
public abstract class InteractableBase : MonoBehaviour {
    public abstract void OnClick();          // 点击响应
    protected void RequireItem(string id);   // 需要某道具才能交互
    protected void PlaySfx(string clip);     // 走 AudioManager
}

// 3. 谜题基类：解开后统一广播事件
public abstract class PuzzleBase : MonoBehaviour {
    public UnityEvent OnSolved;              // 解开时触发（连门、连剧情）
    protected void Solve();                  // 子类判定成功后调用
}

// 4. 场景切换：B 只管调用，不碰加载细节
SceneLoader.GoToRoom("Room_01_Altar");
```

> **铁律**：内容层（B）永远只调用系统层暴露的**公开接口**，不直接读写系统层内部字段。接口若要改，A 先在 `接口契约.md` 记一笔并通知 B。

## 三·五、主角操控系统（点击寻路行走）

> 本作虽是绣湖类，但**主角老人在场景内可行走**（区别于经典绣湖的纯视点）。这是 2026-07-12 敲定的核心设计，详细接口见 `接口契约.md` §7~§10。

**玩家操控 = 四层：**
1. **点击交互**：光标 hover 高亮 → 点击触发。
2. **老人行走（玩家的"移动"）**：点空地 → 老人走过去；点物件 → 老人走到该物 `interactPoint` → 到位后才触发 `OnClick()`。
3. **视点切换**：点门 → `SceneLoader.GoToRoom()` 换房间；点抽屉/信件 → `CloseupView` 弹特写层（不换场景）。
4. **背包**：选中物品 → 光标持物态 → 点目标物（`InventorySystem.SelectedItem` 判断）。

**寻路方案（已定）**：**单屏可走带** —— 每房间一个 `WalkableArea` 圈出地面，老人直线移动并被 clamp 在可走区内，**不绕障碍**。个别房间需要绕障碍再单独升级为路点图。够绣湖类单屏房间用，零第三方依赖。

**系统层新增部件（A 负责）**：`PlayerController`（行走/朝向/动画/深度排序）、`WalkableArea`（可走区）、`CloseupView`（特写）；`InteractableBase` 改为"点击→走到交互点→触发"流程，新增 `interactPoint` / `walkToBeforeInteract` 字段。

**内容层新增义务（B 负责）**：每个房间摆一个 `WalkableArea` 圈地面；每个可交互物摆一个 `interactPoint` 空物体标"老人站哪儿点它"。

**美术新增需求**：老人的行走动画（至少 idle + 左右 walk 两套帧或骨骼动画），这是纯绣湖不需要的，需与美术同步。

## 四、Unity 工程目录结构（Assets 下）

> 规范的目录 = 双人不打架的物理基础。**每个房间一个独立场景文件**，是绣湖类避免场景冲突的关键。

```
Assets/
├── _Project/                 # 我们自己的东西全放这，第三方插件放外面
│   ├── Scenes/
│   │   ├── Boot.unity              # 启动场景（只做初始化，A 负责）
│   │   ├── MainMenu.unity          # 主菜单
│   │   └── Prologue/               # 序章的房间，每间一个 .unity
│   │       ├── P_00_Entrance.unity
│   │       ├── P_01_Hall.unity
│   │       └── ...
│   ├── Scripts/
│   │   ├── Core/            # 系统层（A 的地盘）
│   │   │   ├── Managers/    # GameManager/SceneLoader/AudioManager...
│   │   │   ├── Save/        # SaveSystem/GameState
│   │   │   ├── Interaction/ # InteractableBase/PuzzleBase 基类
│   │   │   └── Dialogue/    # DialogueSystem
│   │   ├── Rooms/           # 内容层（B 的地盘），按章分文件夹
│   │   │   └── Prologue/
│   │   ├── Puzzles/         # 具体谜题脚本
│   │   └── Utils/           # 公共工具（协商后共用）
│   ├── Art/                 # 美术交付放这（美术同学用）
│   │   ├── Backgrounds/
│   │   ├── Sprites/
│   │   ├── UI/
│   │   └── Animations/
│   ├── Audio/               # 音乐/音效
│   ├── Prefabs/             # 预制体（可交互物做成 Prefab 复用）
│   ├── Data/                # ScriptableObject：物品表/对白表/谜题配置
│   └── Fonts/
├── Plugins/                 # 第三方插件（DOTween 等）
└── StreamingAssets/
```

### 命名规范（强制）
- 场景：`章前缀_序号_英文名` → `P_01_Hall`（P=Prologue, O=Old, M=Middle, Y=Young）
- 脚本类名：`PascalCase`，谜题以 `Puzzle_` 结尾，交互物以 `Interact_` 结尾
- Flag/物品 key：`snake_case` 字符串常量集中放在 `Data/Keys.cs`，**禁止散落魔法字符串**
- 美术资源：`bg_房间名`、`spr_物件名`、`ui_控件名`

## 五、版本控制与"防场景冲突"规范（重中之重）

Unity + Git 双人开发最大的坑是 **两人改同一个 `.unity` 或 `.prefab` 文件 → 合并几乎无法解决**。下面是必须执行的规矩。

### 5.1 一次性设置
1. **Git LFS**：美术/音频/大二进制走 LFS（见 `.gitattributes`）。
2. **Unity 强制文本序列化 + 可见 meta**：
   `Edit > Project Settings > Editor`
   - Asset Serialization → **Force Text**
   - Version Control Mode → **Visible Meta Files**
3. **Smart Merge（UnityYAMLMerge）**：配置 Git 用 Unity 自带工具合并场景/预制体（见附录命令），能救一部分场景合并。

### 5.2 铁律（团队约定，写进群公告）
- **一个场景/预制体同一时间只有一个人改**。开工前在协作看板/群里"认领"该场景。
- **每个房间独立场景**，天然把冲突面缩到最小 —— 这就是我们把每间房拆成独立 `.unity` 的原因。
- 系统层公共文件（GameManager 等）**只有 A 改**；B 需要新增全局功能 → 提给 A，别自己动。
- **每天开工先 `git pull`，收工前 `git push`**，别攒一周再合。
- 提交信息带前缀：`[core]`/`[room]`/`[puzzle]`/`[art]`/`[fix]`。

### 5.3 分支策略（轻量）
```
main            # 始终可运行的稳定版，只接受合并
└── dev         # 日常集成分支，两人每天汇入
    ├── feat/core-save        # A 的功能分支
    └── feat/room-entrance    # B 的功能分支
```
- 每人在自己的 `feat/*` 分支干活，一个功能/一个房间一条分支。
- 完成→合进 `dev`→大家 pull dev。里程碑稳定后 `dev` 合 `main` 打 tag（如 `v0.1-prologue`）。
- 小团队可简化：直接两人各推 `dev`，但**必须勤 pull**。

## 八、给后续章节的复用建议
- 序章做出的**系统层 = 全项目地基**，后三章基本只写内容层，产能会显著提升。
- "同一房间三个时代版本"用**同场景 + 按 `CurrentEra` 切换 sprite/物件启用**实现，避免重复建三份场景。
- 每章沿用相同目录/命名/分支规范，新程序员也能快速上手。
